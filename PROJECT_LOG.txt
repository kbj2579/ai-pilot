AI Dogfight Pilot 프로젝트 진행 기록
=====================================
(Claude Code가 작성한 설계/작업 로그. 대회명: AI Pilot 1v1 공중전 강화학습 경진대회)


1. 사용자의 최초 요청
---------------------
- 목적: 내 AI 비행기의 "생존 능력"과 "격추 능력"을 키워서 대회에서 살아남기.
- 방식: 1대1(1v1) 대결. 우선은 임의로 정해진 봇을 상대로 강화학습을 진행하고,
  이후 실제 상대와 겨룬다.
- 우선순위: (1) 내 비행기가 혼자서 추락하지 않는 상태를 먼저 만들고,
  (2) 그 다음에 상대 비행기를 격추시키는 경험을 쌓는다.
- 반드시 읽고 설계에 반영해야 하는 참고자료 2개:
    - "Releasestudent 5 파일 작성·실행 매뉴얼.txt"
      : 학생이 수정 가능한 파일은 student/ 폴더의 5개 파일(my_reward.py,
        my_observation.py, my_curriculum.py, my_train.py, my_submission.py)
        뿐이며, 본체 학습 루프(train_rllib.py, train_curriculum.py,
        scripts/run_experiment.py, src/dogfight/*)는 손대면 안 된다는 계약을
        설명하는 공식 매뉴얼.
    - "보상 관련 설계 체크리스트.txt"
      : World-NED vs Body-frame, ATA(추격각)/AA(피추격각)/HCA(진행방향차)/
        WEZ(무기교전범위)/Closure(접근속도)/Energy(에너지 우위) 등 공중전
        보상 설계 이론과 실패 패턴(예: ATA만 보상하면 계속 뱅뱅 도는 정책이
        학습됨) 체크리스트.
- 추가 요청: Claude Code/에이전트/하네스 엔지니어링을 활용해 실시간으로
  코드를 수정하고 강화학습을 진행하며, 가능하면 원격(폰)에서 진행상황을
  보거나 제어하고 싶다.

------------------------

    - src/dogfight/    : 본체 환경/보상/종료판정/커리큘럼/RLlib 연동 코드

- Rule.xml, PropertySheets/, Windows/ : 대회 인프라 관련 파일.


3. 세운 계획 (전체 요지)
-------------------------
Phase 1 — 생존: 내 비행기가 혼자서도 추락/실속하지 않고 안정적으로 비행하도록
학습. Phase 2 — 격추: Phase 1 위에서 커리큘럼 학습으로 난이도를 점진적으로
올려 (고정 표적 → loiter 표적 → autopilot 표적 → behavior_tree 봇), 상대를
격추하는 경험을 쌓는다.

사용자가 확정한 결정 3가지:
  - RL 알고리즘은 PPO부터 시작 (GPU가 GTX 1660 Super 6GB로 제한적이라 SAC와
    동시에 돌리지 않음).
  - 원격/폰 모니터링은 지금은 생략 (로컬 대시보드만 사용, 보안 설정은 나중에).
  - 학습 캠페인은 "최대 자율 진행" — Claude Code가 여러 날에 걸쳐 백그라운드로
    학습을 계속 돌리고, 커리큘럼 진급/보상 조정까지 스스로 판단해서 진행하며,
    중간 승인 없이 주기적 요약만 보고한다. (사용자가 이후 "나 자고올테니까
    너가알아서해봐"라고 재확인함)

설계한 보상 함수(student/my_reward.py) 구조:
  - 전 구간 공통: step_penalty(시간 압박).
  - Phase 1(안전) 컴포넌트: survival_bonus, altitude_safety(최저고도 페널티),
    attitude_stability(과도한 롤/피치 페널티), speed_safety(실속 방지, 현재는
    비활성 — 어떤 로그에도 KCAS가 노출되지 않아 정밀 보정 불가, 추후 재검토).
  - Phase 2(전투) 컴포넌트: pursuit_shaping(ATA×거리 그라디언트),
    aspect_shaping(cos(AA), 반드시 pursuit와 함께 사용 — 아니면 "뱅뱅 도는"
    퇴화 정책 위험), closure_shaping(HCA로 게이팅된 접근속도 보상, WEZ 근처에서
    게인 축소), wez_bonus(이산+연속 보너스), enemy_wez_penalty(상대 WEZ 안에
    있을 때 페널티, ATA 인자를 뒤집어 호출해서 계산), damage(피해량 차이),
    terminal(승/패/무 + 헤드온 가드 실패 페널티).
  - 커리큘럼 단계마다 reward_overrides로 가중치만 교체 (구조는 고정).

설계한 커리큘럼(student/my_curriculum.py), 총 7단계(0~6):
  0 flight_survival        (fixed 표적)         — 순수 비행 안정화
  1 target_orientation     (fixed 표적)         — 표적 지향/접근
  2 wez_approach            (loiter 표적)        — WEZ 진입
  3 pursuit_autopilot       (autopilot 표적)     — 이동 표적 추격
  4 merge_headon_bt_easy    (behavior_tree 봇)   — 헤드온 머지 (alpha~75도)
  5 merge_headon_bt_hard    (behavior_tree 봇)   — 헤드온 머지 (alpha~165도)
  6 full_dogfight           (behavior_tree 봇)   — 자유 교전 (최종 단계)
본체 참고 커리큘럼(15단계)보다 컴퓨팅 예산에 맞게 압축한 버전이며, 그대로
베끼지 않고 독자적으로 설계함.

4. 실제로 진행한 작업 (시간순)
-------------------------------
1) 저장소 전체 구조 및 두 매뉴얼 txt 완독 (Explore 서브에이전트 병렬 활용).
2) 계획 수립 → 사용자 승인 (RL 알고리즘/원격 모니터링/자율성 수준 확정).
3) student/my_reward.py, student/my_curriculum.py 작성.
4) experiments/student_stage0_ppo_mlp.yaml 작성 후 Stage 0 단독 검증:
   dry-run → 1-iteration smoke test → 200-iteration 본학습.
   결과: crash_rate=0.0, timeout_rate=1.0 전 구간 유지 — Phase 1 목표 달성.
5) 이 과정에서 발견하고 고친 버그:
   - episode_step_limit 단위 착각: raw tick이 아니라 RL 매크로스텝
     (step_ratio=6 적용 후) 단위였음. 7단계 전부 의도한 것보다 6배 긴
     에피소드가 될 뻔한 것을 발견하고 전부 재계산해서 수정.
   - 본체 training_record.py의 describe_reward()가 "low_altitude_penalty"
     키를 무조건 요구해서(내 보상 설계엔 없는 키) 학습 후 기록 저장이 깨짐
     → 실제로는 안 쓰지만 호환용 더미 키로 추가해서 해결.
   - env_config.reward YAML 블록이 있으면 student.my_reward가 조용히
     무시되는 함정 발견 → YAML에서 해당 블록을 아예 빼는 방식으로 회피.
   - "wez_approach"라는 정확한 이름일 때만 OR 진급조건이 적용되는 하드코딩
     로직을 소스에서 확인하고 커리큘럼 설계에 반영.
6) train_curriculum.py로 7단계 전체 자동 학습 시작 (curriculum_v1).
   → 워커 4개 중 1개가 학습 시작 직후부터 "FDM Update Fail"(비행 시뮬레이터
     갱신 실패)에 2만 번 이상 연속으로 갇히는 문제 발견. 학습 데이터 1/4
     손실 + 지표 집계(win/crash/timeout_rate)가 전부 "n/a"로 깨짐.
   → 프로세스 종료, Ray 정리, 워커 수 4→3으로 줄여서 재시작(curriculum_v2).
   → 그런데도 동일하게 runner:1이 처음부터 같은 문제에 갇힘.
   → 원인 분석: 본체 참고 커리큘럼의 Stage 0은 위치 랜덤화 radius=0.0을
     쓰는데, 내 Stage 0은 radius=100.0을 써서 초기 위치를 흔들고 있었음.
     격리된 Stage-0 단독 테스트(랜덤화 없음)는 전혀 문제가 없었던 반면,
     랜덤화를 켠 커리큘럼 실행에서만 문제가 재현된 것으로 보아, 무작위
     초기 자세/위치가 드물게 JSBSim 비행 모델을 되돌릴 수 없이 망가뜨리고,
     한번 망가진 워커는 그 이후 리셋에서도 계속 실패하는 것으로 추정.
   → 대응: Stage 0의 랜덤화 radius를 100→20(작은 양수)으로 낮추고 재시도
     (curriculum_v3). 그런데 radius=0.0을 그대로 썼던 부분(본체 참고
     커리큘럼 값을 실수로 복사)에서 완전히 다른 버그 발견:
     add_random_init_position()이 `self.np_random.integers(0, radius)`를
     radius=0일 때 가드 없이 호출해서 `ValueError: high<=0`으로 워커 전체가
     즉시 크래시(access violation). 이건 본체 코드 자체의 잠재적 버그로
     보임(수정 불가 파일). radius를 0이 아닌 작은 양수(20.0)로 바꿔서 해결.
   → curriculum_v4로 재시작: Stage 0(radius=20)은 200 iteration 전부 정상
     완주(crash_rate=0.0 계열 재확인). 그런데 Stage 1로 넘어가는 weight
     restore 단계에서 본체 train_curriculum.py가 콘솔에 유니코드
     체크마크(✓/✗)를 출력하려다 이 PC의 한글 Windows 코드페이지(cp949)가
     이를 인코딩 못해 UnicodeEncodeError로 전체 프로세스가 죽음. 파일 수정
     없이 PYTHONIOENCODING=utf-8, PYTHONUTF8=1 환경변수를 추가하고
     --resume으로 재시작.
   → 인코딩 문제는 해결됐으나, Stage 1(radius=150, r_pitch만 6으로 축소)에서
     "FDM Update Fail" 락업이 또 재발(이번엔 다른 워커). r_pitch를 줄인 것만
     으로는 부족했고 radius 자체가 원인임을 재확인. 최종적으로 전체 7단계의
     radius를 전부 20.0으로 통일(안전이 확인된 Stage 0 수준)하고 --resume으로
     재시작. (참고: 중간에 PC가 한 번 꺼졌다 켜지는 일이 있었음 — 모든 학습
     프로세스가 사라져서 재시작함.)
7) 사용자가 학습을 일시 취소하고 "다른 사람과 스크림(연습 대결) 환경 구축"을
   요청. curriculum_v4는 Stage 3 진행 중 상태로 안전하게 중단(체크포인트/
   curriculum_state.json에 진행상태 보존, 나중에 --resume으로 재개 가능).
   스크림 방식은 실제 Unreal 교전서버(BattleServer) 연결 방식으로 결정.
   준비 과정에서 발견/처리한 것:
   - Windows/DogFightViewer.exe가 실제 BattleServer와 같은 것인지 저장소
     안에서는 확인 불가(문자열 검색에도 관련 흔적 없음, 크기도 168KB로
     작음) → Discord 공지에 따르면 BattleServer는 팀 공유 구글드라이브
     (Update -> BattleViewer)에서 별도로 받는 실행파일이 맞는 것으로 결론.
   - DogFightEnv/Release/ 루트에 BT 모드에 필요한 Rule XML(기본값
     "Rule_forTraining.xml")이 아예 없다는 것을 발견(학습 중엔 필요 없어서
     지금까지 몰랐음). 저장소 루트의 Rule.xml(스텁)을 복사해서
     Release/Rule_forTraining.xml로 생성.
   - student/my_submission.py를 스크림용으로 수정: MODE="bt"로 전환(아직
     완성된 RL 모델이 없으므로 연결 테스트 목적), SERVER_IP를
     "127.0.0.1"(로컬 테스트 기본값)로 바꾸고 상대 IP로 교체하도록 TODO
     주석 추가, AI_TYPE도 모드에 맞게 자동 전환되도록 수정.
   - run_unreal_inference.py --mode bt와 student/my_submission.py 둘 다
     실제 서버 없이 로컬(127.0.0.1)로 스모크 테스트: DLL 로드, Rule XML
     활성화, UDP 소켓 오픈까지 전부 정상 동작 확인(서버가 없어 패킷
     자체는 못 받았지만 클라이언트 쪽 코드 경로는 문제없음).


5. 현재 상태
-------------
- Phase 1(생존)은 격리된 단독 실행과 커리큘럼의 Stage 0 실제 실행 모두에서
  검증 완료(crash_rate=0.0, 200 iteration 전부 완주).
- 7단계 전체 자동 커리큘럼 파이프라인은 서로 다른 원인의 크래시를 4차례
  겪으며 디버깅 중(FDM 락업 2회, numpy ValueError 1회, 유니코드 인코딩 1회).
  radius를 전 단계 20.0으로 통일 + UTF-8 인코딩 강제 설정 후 재개해서
  Stage 0/1/2 완료, Stage 3 진행 중(229/250)이었으나, 사용자 요청으로
  일시 취소함. 진행상태는 artifacts/curriculum/team01/curriculum_v4/에
  전부 보존되어 있어 --resume으로 언제든 이어서 진행 가능.
- 랜덤화(ownship_randomization)를 켜두면 radius를 아무리 줄여도 아주
  낮은 확률로 워커 하나가 영구적으로 "FDM Update Fail"에 갇히는 현상은
  본체 코드 자체의 특성으로 보이며 완전히 없앨 수는 없음을 확인(자세한
  내용은 Claude 메모리 project_ai_dogfight_fdm_lockup 참고). 학생이 수정
  가능한 파일로는 근본 해결이 불가능해서, 감지되면 재시작하는 운영
  전략으로 대응하기로 함.
- 스크림 환경: student/my_submission.py가 MODE="bt"로 로컬(127.0.0.1)
  기준 정상 동작 확인 완료. 실제 상대와 붙으려면 (a) 상대측이 BattleServer
  실행파일(팀 공유 구글드라이브에서 확보 가능한 상태)을 열고 "OpenServer"
  누른 뒤, (b) 그 PC의 IP를 알려주면 my_submission.py의 SERVER_IP를 그
  값으로 바꿔서 실행하면 됨. 아직 완성된 RL 모델이 없어 BT 모드로 우선
  연결 테스트만 가능한 상태.
- 사용자 결정에 따라 대회 서버 제출(student/my_submission.py) 등 되돌리기
  어려운 작업은 하지 않고, 학습 진행 상황만 계속 갱신한다.
- 대회 운영진 Discord에서 공지된 환경 패치(체크포인트 저장 정책 분리,
  custom observation bundle 복원 hotfix, 2026-05-27 / 2026-07-08자)는
  현재 저장소에 아직 반영되어 있지 않음을 확인했으며, 사용자 결정에 따라
  지금은 보류하고 custom observation을 실제로 쓰게 될 때 공식 Release
  zip으로 제대로 반영하기로 함.

  run_local_dogfight.py로 양쪽에 서로 다른 bundle을 지정해 즉시 대결 가능,
